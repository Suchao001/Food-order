# Technical Design Document: Cafe & Beverage POS System Extension

## 1. Problem Statement
ปัจจุบันร้านค้าใช้ระบบสั่งอาหารออนไลน์ (`Food-order`) ที่พัฒนาด้วย Nuxt 3 และ PostgreSQL ในการรับสั่งอาหาร แต่เนื่องจากร้านกำลังขยายไปทำคาเฟ่เพิ่มเติม (มีทั้งเมนูเครื่องดื่มและขนม) จึงต้องการต่อยอดระบบนี้ให้สามารถสั่งเครื่องดื่มและขนมได้ด้วย 
เป้าหมายคือ:
1. ลูกค้าหรือผู้ใช้สามารถสั่งอาหาร เครื่องดื่ม และของหวานได้พร้อมกันใน **บิลเดียว (Combined Receipt)**
2. ระบบจัดการหลังบ้านสามารถ **แยกสลัดออร์เดอร์** ออกจากกันอย่างเด็ดขาด โดยออร์เดอร์ที่ไม่ใช่อาหาร (เช่น กาแฟ) จะไม่ไปรกบนหน้าจอห้องครัว แต่จะแยกไปแสดงที่หน้าจอบาริสต้า (เครื่องดื่ม) แทน
3. มีระบบ **Dashboard** ที่แยกรายงานและวิเคราะห์ยอดขายตามแต่ละแผนก (อาหาร / เครื่องดื่ม / ขนม) อย่างชัดเจน

---

## 2. User Persona & Target Audience
* **ลูกค้า / แคชเชียร์ (Customer / Cashier)**: สั่งเมนูผ่านหน้าแท็บเล็ต สะดวกรวดเร็วในการกดเปลี่ยนหมวดหมู่ และฝั่งหน้าจอ Cashier / จุดรับออร์เดอร์รวม จะเห็นรายการทั้งหมดทั้งอาหารและเครื่องดื่มในบิลเดียวกันเพื่อคิดเงิน โดยมีการแยกโซนการแสดงผล (เช่น อาหาร / เครื่องดื่ม) ออกจากกันอย่างชัดเจนบนหน้าจอ เพื่อให้อ่านง่าย
* **แม่ครัว (Chef / Mother)**: ประจำหน้าจอครัว (`/kitchen`) ทำอาหารเฉพาะของห้องครัวเท่านั้น ไม่ต้องการเห็นเครื่องดื่มบนหน้าจอ
* **บาริสต้า (Barista)**: ประจำหน้าจอเคาน์เตอร์กาแฟ (`/barista`) ดูเครื่องดื่มและขนมที่ต้องทำ ไม่ต้องการเห็นรายการข้าวบนหน้าจอ
* **ผู้จัดการ / เจ้าของร้าน (Admin / Manager)**: ตรวจดูยอดขายรวมของร้านทางหน้า Dashboard และวิเคราะห์แยกย่อยรายได้จากแผนกกาแฟ เทียบกับอาหาร

---

## 3. System Requirements
1. **Multi-category Ordering UI**: หน้าจอการสั่งซื้อหลักเพิ่มเมนูตัวเลือกประเภท (Tabs Filter) เพื่อดู "อาหาร" / "เครื่องดื่ม" / "ของหวาน"
2. **Conditional Options Modal**: ป๊อปอัปเลือกตัวเลือกเสริมของเครื่องดื่ม เช่น ความเย็น (ร้อน, เย็น, ปั่น), ความหวาน (0%, 25%, 50%, 100%), ประเภทนม (นมสด, นมโอ๊ต) และวิปครีม
3. **Speech Synthesis Extension**: ระบบ TTS อ่านออร์เดอร์ที่ปรับแต่งเพิ่มเติมได้ครบถ้วน เช่น *"อเมริกาโน่เย็น หวานน้อย 1 แก้ว"*
4. **Order Items Status Logic**: อัปเกรดฐานข้อมูลให้ติดตามความคืบหน้าระดับรายการย่อย (`order_items.status`) เพื่อแยกจอรันคิวการผลิตของครัวและบาร์น้ำ
5. **Separate Production Dashboards**:
   * หน้าจอห้องครัว (`/kitchen`) แสดงเฉพาะรายการที่เกี่ยวข้องกับแผนกครัว
   * หน้าจอบาริสต้า (`/barista`) แสดงเฉพาะรายการที่เกี่ยวข้องกับแผนกบาร์เครื่องดื่ม
6: **Unified Analytics Dashboard**: แดชบอร์ดสรุปรายได้ที่แจกแจงแยกสัดส่วนตามรายแผนก
7. **Cashier / Receipt Visual Separation**: หน้าจอสำหรับแคชเชียร์และใบเสร็จสรุปยอด จะต้องแสดงผลรายการทั้งหมด (ทั้งอาหาร เครื่องดื่ม และขนม) ในบิลเดียว แต่ในฝั่งหน้าจอแสดงผลของแคชเชียร์ จะทำการแยกโซนหรือแบ่งสัดส่วนการแสดงรายการอาหารและเครื่องดื่มออกจากกันอย่างชัดเจน และสามารถออกแบบเป็น **Tab เพื่อสลับการดูข้อมูลได้** (เช่น แถบ "ดูทั้งหมด" / แถบ "เฉพาะรายการอาหาร" / แถบ "เฉพาะรายการน้ำ") เพื่อช่วยให้เช็คของง่าย ไม่สับสนขณะรับออร์เดอร์และหยิบเสิร์ฟ

## 4. Tech Stack & Dependencies
* **Core**: Nuxt 3 (v3.13.0) & Vue 3 (v3.5.26)

## 6. Data Schema & Type Definitions

เราจะรันสคริปต์ SQL เพื่อปรับปรุงฐานข้อมูลดังนี้:

```sql
-- เพิ่มประเภทสำหรับแผนกเตรียมอาหาร
CREATE TYPE preparation_dept AS ENUM ('Kitchen', 'Barista', 'Bakery');

-- เพิ่มประเภทสถานะของรายการย่อย
CREATE TYPE item_status AS ENUM ('Pending', 'Preparing', 'Ready', 'Cancelled');

-- ตารางประเภทเมนูหลัก
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL, -- 'Food', 'Beverage', 'Dessert'
    icon VARCHAR(10) NOT NULL, -- '🍜', '☕', '🍰'
    display_order INT DEFAULT 0
);

-- ปรับปรุงตาราง menus
ALTER TABLE menus ADD COLUMN category_id INTEGER REFERENCES categories(id) ON DELETE SET NULL;
ALTER TABLE menus ADD COLUMN dept preparation_dept NOT NULL DEFAULT 'Kitchen';

-- ปรับปรุงตาราง order_items
ALTER TABLE order_items ADD COLUMN status item_status DEFAULT 'Pending';
```

## 7. Scope Boundaries

### ✅ What We WILL Do (MVP Boundary)
* การอัปเกรดฐานข้อมูลโครงสร้างเดิมให้รองรับหมวดหมู่ และแผนกรับผิดชอบ
* หน้าสั่งซื้อเครื่องดื่มที่มีความยืดหยุ่นสูง (รองรับความหวาน ความเย็น ท็อปปิ้ง และนมอัลเทอร์เนทีฟ)
* ระบบส่งสัญญาณเสียงทบทวนออร์เดอร์ (Text-to-Speech) รองรับการสั่งกาแฟ
* หน้าจอแยกการเตรียมอาหารสำหรับคนปรุงกาแฟ บาริสต้า (`/barista`) โดยไม่ไปรบกวนหน้าจอห้องครัว (`/kitchen`)
* รายงานแดชบอร์ดรายได้รวมที่ดึงสัดส่วนยอดขายแยกของกาแฟ ขนม และอาหารได้

### ❌ What We WILL Not Do (Out of Scope / Future Work)
* ระบบนับสต็อกวัตถุดิบ (เช่น เมล็ดกาแฟ, นมสด, นมโอ๊ต)
* การพิมพ์สติกเกอร์บาร์โค้ดแปะแก้วด้วยความร้อนผ่านเครื่องพิมพ์ (เก็บไว้ทำในเฟสถัดไป)
* ระบบสมาชิกสะสมแต้ม หรือโปรโมชั่นสะสมแก้วแถมแก้ว

## 8. Failure Points & Edge Cases
* **Edge Case: ลูกค้ายกเลิกบิลขณะทำอาหาร**: หากยกเลิกตาราง `orders` ต้องทำการยกเลิกสถานะของทุก `order_items` ในทุกแผนกด้วย และส่งเสียงเตือนทั้งทางฝั่งครัวและบาร์น้ำ
* **Edge Case: การจัดการสถานะออร์เดอร์รวม**: หากบาริสต้าทำกาแฟเสร็จเป็นคนแรก แต่ห้องครัวยังทำข้าวไม่เสร็จ ตัวสัญญาณเรียกรับสินค้าหลักของบิลนั้นจะไม่ทำงานจนกว่าแผนกสุดท้ายจะกดยืนยันสำเร็จ

---

## 9. Known Unknowns
* โหมดการส่งสัญญาณ Realtime ระหว่างฝั่งหน้าจอทำเครื่องดื่ม/อาหาร: ปัจจุบันใช้ระบบดึงข้อมูลแบบ polling ทุก 30 วินาที หรือต้องการให้วางระบบ WebSocket/SSE (Server-Sent Events) ทันทีในเฟสนี้เลยหรือไม่? *(ในเบื้องต้นจะแนะนำให้คงสถาปัตยกรรมเดิมไว้ก่อนเพื่อความรวดเร็วในการพัฒนา)*

---

## 10. Scale & Performance Limits
* จำกัดจำนวนการแสดงผลในหน้าจอออร์เดอร์ปัจจุบันของห้องครัวและบาร์ ไม่เกิน 50 ออร์เดอร์ล่าสุดที่ยังทำไม่เสร็จ เพื่อการประมวลผลและการโหลดหน้าจอที่รวดเร็วบนแท็บเล็ตสเปกต่ำ

---

## 11. Expected Outcomes & Acceptance Criteria
1. สามารถเข้าหน้า `/order` และกดสั่งเมนูเครื่องดื่มที่มีตัวเลือกเสริม (เช่น หวาน 50% และนมโอ๊ต) ได้ โดยระบบคำนวณราคาได้ถูกต้อง
2. เมื่อกดสั่งอาหาร + น้ำ รายการอาหารแสดงใน `/kitchen` เท่านั้น ส่วนกาแฟแสดงใน `/barista` เท่านั้น
3. หน้าจอแดชบอร์ดแสดงผลกราฟสัดส่วนรายได้แยกตามแผนก (`Kitchen` vs `Barista`) ได้ถูกต้องตามข้อมูลในฐานข้อมูลจริง

## 12. แผนการพัฒนาแบบแบ่งเป็นเฟส (Phase-by-Phase Plan)

เพื่อการทำงานอย่างมีขั้นตอนและทดสอบได้ง่าย ขอเสนอแผนการแบ่งเฟสการทำงานดังนี้:

### 🚀 Phase 1: Database Migration & Config Setup
* [ ] เขียนสคริปต์สำหรับการ Migrations โครงสร้างของฐานข้อมูล PostgreSQL (เพิ่ม `categories`, คอลัมน์ `dept` และ `status` ใน `order_items`)
* [ ] รันสคริปต์สร้างข้อมูลจำลอง (Seed Data) หมวดหมู่ ข้อมูลเมนูเครื่องดื่ม (เช่น Latte, Americano) ข้อมูลขนมเค้ก และรายการตัวเลือกเสริมของเครื่องดื่มลงฐานข้อมูลจริง
* [ ] อัปเดต Model / Types และ Connection ให้รองรับฟิลด์ใหม่ทั้งหมด

### 🎨 Phase 2: Core Admin Dashboard Updates
* [ ] แก้ไขหน้าจอจัดการรายการร้านอาหารในฝั่ง Admin เพื่อให้สามารถเพิ่ม/แก้ไข เมนู และจัดระเบียบแผนกการทำงาน (`dept`), หมวดหมู่ (`category_id`)
* [ ] สร้าง API จัดการ CRUD หมวดหมู่และฟิลด์ใหม่

### 🛍️ Phase 3: Client Order UI & Custom Cafe Options
* [ ] เพิ่มตัวกรองสลับแถบแสดงผลประเภทเมนู (อาหาร เครื่องดื่ม ขนม) ในหน้าสั่งซื้อ
* [ ] พัฒนาโมดอลสั่งอาหารให้รองรับการเลือกตัวเลือกเฉพาะของเครื่องดื่มและขนมได้อย่างสวยงามและคุมเงื่อนไขการเลือก (เช่น สั่งลาเต้ปั่นต้องบวกเงิน สั่งชาร้อนไม่มีตัวเลือกความเย็น เป็นต้น)
* [ ] เชื่อมระบบเสียงอ่านออร์เดอร์ให้สนับสนุนความซับซ้อนของตัวเลือกน้ำและอาหาร

### 🧑‍🍳 Phase 4: Production Order Routing Screens
* [ ] ปรับหน้าจอครัวเดิม (`/kitchen`) ให้ฟิลเตอร์กรองเฉพาะรายการอาหาร
* [ ] สร้างหน้าจอบาริสต้าใหม่ (`/barista`) ที่แสดงเฉพาะรายการกาแฟและขนมหวาน
* [ ] พัฒนาหน้าแสดงผลของจุดแคชเชียร์และใบเสร็จสรุปออร์เดอร์รวม ให้แสดงผลแยกบล็อกสินค้า หรือสลับการแสดงผลผ่านแท็บ (Tabs: ดูทั้งหมด / อาหาร / เครื่องดื่ม) เพื่อให้พนักงานที่จุดรับออร์เดอร์ตรวจสอบความพร้อมได้สะดวกรวดเร็ว
* [ ] พัฒนา API อัปเดตสถานะของไอเทมแยกรายชิ้น พร้อม Logic ออโต้อัปเดตตารางบิลใหญ่

### 📊 Phase 5: Multi-department Sales Dashboard
* [ ] พัฒนาคิวรีฝั่งหลังบ้านสำหรับดึงยอดขายแยกกลุ่มสินค้าและแจกแจงรายได้
* [ ] อัปเดตหน้า Dashboard เพื่อแสดงยอดขายรายแผนก สัดส่วนการขาย และวิเคราะห์รายได้รายวันของแต่ละแผนก
